### UAT-CLI-004 — initial FAIL, fixed mid-run

**Symptom:** `vault run --profile <any> -- <cmd>` errored with
`failed to load secret for ai.zyr1.vault/credential:<id>:api_key / Caused by: The user
name or passphrase you entered is not correct.` Affected ALL credentials (uat-test +
pre-existing work-main), including items with `-A` (allow-all) ACL.

**Root cause:** `SecKeychain::disable_user_interaction()` at
`crates/vault-secrets/src/keychain.rs:141` returns `errSecAuthFailed` on macOS 15 +
`security-framework` 3.7.0 even when the ACL permits silent read by the calling
binary. The call was added during hardening to avoid silent GUI prompts in automated
contexts but regressed baseline `vault run` functionality.

**Fix (applied mid-run, not yet committed at time of this run-log):** remove the
`disable_user_interaction()` guard from `SecretStore::get` and the now-unused
`SecKeychain` import. Follow-up to find a non-regressing replacement (e.g.
`SecItemCopyMatching` with `kSecUseAuthenticationUINone`) tracked in ROADMAP.

**Regression check:** `cargo test --workspace` still reports 73 passing, 0 FAILED
after the fix → UAT-SEC-002 unaffected.

**Ancillary finding — VAULT_LEASE_TOKEN shape:** value emitted was two concatenated
UUIDs without a separator (`<uuid1><uuid2>`, 72 chars). Passes the UAT-CLI-004
"uuid-shape" criterion loosely but warrants a follow-up pass to confirm intent.

**Ancillary finding — `security add-generic-password -w` (stdin-less form):**
diagnostic test (`echo -n "value" | /usr/bin/security add-generic-password -w`)
shows `security` falls back to `/dev/tty` prompt regardless of stdin piping. The
`put_with_access` flow in `keychain.rs` therefore reads passwords from the
terminal, not the intended stdin pipe. Not a UAT failure (UAT-CLI-002 still passes
via interactive fallback) but the code comment and `CLAUDE.md` gotcha both need
correction, and the secret-in-process-list protection the flow claims to provide
is not actually in place.
